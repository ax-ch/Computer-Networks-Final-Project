ISSUE #001
------------------------------------------------------------------------
CONTEXT: 
Phase 0 - Workspace & Repository Initialization

SYMPTOM: 
During the initial 'git push' to sync the provided HTML assets, the terminal threw a warning: "File assets/osi.mp4 is 54.67 MB; this is larger than GitHub's recommended maximum... GH001: Large files detected."

ROOT CAUSE: 
GitHub has a soft limit of 50 MB per file, triggering a Git LFS (Large File Storage) warning prompt. However, the hard limit blocking uploads is 100 MB. 

RESOLUTION: 
Verified that the file push actually succeeded despite the warning. Decided to bypass Git LFS configuration to avoid overcomplicating the repository setup, as the file size (54.67 MB) remained safely under the strict 100 MB block limit.

ISSUE #002
------------------------------------------------------------------------
CONTEXT: 
Phase 1 - Web Server Setup (TCP Socket Testing)

SYMPTOM: 
When testing the initial TCP web server at `localhost:8000` via a web browser (Firefox), the expected dummy text ("Hello from TCP Server!") flashed briefly on the screen, followed immediately by an error page stating: "localhost:8000 sent back an empty page."

ROOT CAUSE: 
1. Missing Headers: The initial socket response lacked a `Content-Length` header. Because the socket was closed immediately after sending, the browser assumed the connection crashed mid-transmission.
2. Ghost Requests: The browser automatically sent a secondary background request for `GET /favicon.ico`. The server blindly responded to this image request with the same HTML text string, confusing the browser's parser.

RESOLUTION: 
Updated the `handle_tcp_client` function in `webserver.py` to construct a fully compliant HTTP/1.1 response. Explicitly calculated and added the `Content-Length` header based on the byte size of the payload, and added a `Connection: close` header to cleanly signal the end of the transmission to the browser.

ISSUE #003
------------------------------------------------------------------------
CONTEXT: 
Phase 1/2 - Server Testing & Port Binding

SYMPTOM: 
When launching webserver.py, the terminal threw an OSError: "[WinError 10048] Only one usage of each socket address (protocol/network address/port) is normally permitted" specifically crashing on the udp_socket.bind() function for port 9000.

ROOT CAUSE: 
A previous execution of webserver.py was terminated abruptly. While the main thread closed, the daemonized UDP server thread remained active in the background as an orphaned process, keeping Port 9000 bound and locked.

RESOLUTION: 
Used the Windows command line tool `taskkill /F /IM python.exe /T` to force-terminate all hanging Python processes, successfully freeing up Port 9000 for the next execution.
